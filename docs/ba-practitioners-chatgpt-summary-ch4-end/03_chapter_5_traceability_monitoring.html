# Chapter 5 - Traceability and Monitoring (Chương 5 - Truy vết và giám sát)

Scope note: Chapter 5 explains how requirements stay connected after they are written. Traceability links business needs, requirements, solution components, tests, decisions, defects, and changes so that teams can monitor scope, manage baselines, and make informed impact decisions.

Learning map (Bản đồ học tập)

  • Traceability links needs, requirements, decisions, design, testing, defects, and change.
  • A useful traceability matrix is maintainable, not merely comprehensive.
  • Baselines provide a control point; they do not mean requirements can never change.
  • Impact analysis should cover product, project, business analysis, and stakeholder consequences.
  • Defect handling still needs traceability because a fix can become scope change.

# 5.2 Traceability (Truy vết)

Source page anchor: p. 138 · direct anchor

For a project team, Traceability connects requirement information across the life cycle. The practical purpose is to know why a requirement exists, what it affects, how it will be verified, and what changes if it moves.

Use it when

Use traceability for impact analysis, coverage checks, approval readiness, release planning, test planning, audit needs, and stakeholder questions about scope.

How to apply

Define a minimal set of links and attributes, assign ownership, update links at known process points, and review gaps before baselines and releases.

Practical example

A healthcare software team links a provider-search requirement to the business objective, stakeholder request, search service configuration, test cases, and acceptance evidence. When a stakeholder requests a ranking change, the team can see which tests, documentation, and release notes are affected.

Common mistakes / anti-patterns

  • Creating links once and never maintaining them
  • Tracking too many attributes without ownership
  • Using traceability only for audits instead of daily decisions

Checklist

  • Link each requirement to its source or rationale
  • Connect requirements to tests or acceptance evidence
  • Identify owner, status, priority, and version where useful
  • Review trace gaps before approval or release
Trace directionMeaningExample use
BackwardFrom requirement to source or business needExplain why a feature exists
ForwardFrom requirement to design, build, test, and releaseCheck coverage and readiness
HorizontalAcross peer requirements, rules, interfaces, or dependenciesFind conflicts and coordination needs
StatusAcross requirement life cycle statesMonitor approval, implementation, testing, and defects

Key takeaway: Use Traceability when it helps the team make a clearer decision, reduce uncertainty, or protect business value.

# 5.8 Managing Changes to Requirements (Quản lý thay đổi yêu cầu)

Source page anchor: p. 148 · direct anchor

For a project team, Managing Changes to Requirements keeps requirement evolution from becoming uncontrolled scope drift. Change is not automatically bad; unmanaged change is the problem because it hides cost, risk, conflicts, and downstream work.

Use it when

Use change analysis for new stakeholder requests, defect corrections that alter approved behavior, regulatory updates, technical constraints, and benefit-driven enhancements.

How to apply

Trace the request to affected requirements, assess product and project impacts, identify options, document risks, and recommend a decision path.

Practical example

A late request adds a new notification rule. Impact analysis shows changes to requirements, mobile UI copy, API behavior, automated tests, and user training. The recommendation includes schedule and risk options.

Common mistakes / anti-patterns

  • Approving changes without impact analysis
  • Treating every defect fix as free scope
  • Hiding trade-offs in vague recommendations

Checklist

  • Identify affected requirements and artifacts
  • Estimate BA, delivery, test, and operational impact
  • Show risks and options
  • Record the decision and update trace links

Key takeaway: Use Managing Changes to Requirements when it helps the team make a clearer decision, reduce uncertainty, or protect business value.
