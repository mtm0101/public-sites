# Chapter 4B - Analysis, Models, Documentation, and Agreement (Chương 4B - Phân tích, mô hình, tài liệu hóa và đồng thuận)

Scope note: This page follows the second half of Chapter 4. After information is elicited, the work shifts to analysis: refining raw information, selecting models, documenting solution requirements, prioritizing, validating, verifying, obtaining approval, and resolving conflicts.

Learning map (Bản đồ học tập)

  • Analysis makes raw input coherent, testable, and useful for solution definition.
  • Models expose gaps and conflicts that prose alone often hides.
  • Requirement documentation should match the project life cycle and governance need.
  • Validation asks whether the right need is represented; verification asks whether the requirement is well formed.
  • Conflict resolution benefits from structured techniques when discussion alone stalls.

Source page anchor: p. 88 · direct anchor

In practice, Analyze Requirements addresses transforming elicited information into coherent, testable, and solution-relevant requirements. In a project setting, this topic helps the analyst convert uncertain information into a form that supports decisions, collaboration, and delivery.

Use it when

Use this topic when the team needs clearer understanding, stronger stakeholder alignment, or a more reliable basis for future work.

How to apply

Clarify the business reason, select the right participants, produce an artifact that answers a real question, and confirm the result with stakeholders.

Practical example

Common mistakes / anti-patterns

  • Assuming stakeholder statements are complete
  • Skipping review with affected parties

Checklist

# 4.10 Model and Refine Requirements (Mô hình hóa và tinh chỉnh yêu cầu)

Source page anchor: p. 89 · direct anchor

The practitioner should treat this as Model and Refine Requirements converts complex requirement information into a representation that people can inspect. Models are not decoration; they are thinking tools that expose missing actors, inconsistent terms, hidden business rules, unclear boundaries, and unrealistic assumptions.

Use it when

Use a model when prose is becoming long, ambiguous, or hard to validate. A model is also appropriate when stakeholders disagree because it forces assumptions to become visible.

How to apply

Select the smallest model that answers the decision question, draft it from elicited information, review it with the right stakeholders, and translate discoveries into requirement updates.

Practical example

A user says, "providers need better search." A context diagram shows external data sources, a feature model separates filters from ranking, and a decision table clarifies when fallback results should appear.

Common mistakes / anti-patterns

  • Selecting a model because it looks impressive rather than because it answers a question
  • Failing to validate diagrams with stakeholders
  • Letting models become outdated while requirements change

Checklist

  • Name the decision the model supports
  • Choose a model fit for the audience
  • Validate the model with stakeholders
  • Update requirements based on gaps found

ProcessHow does work move from trigger to outcome?Process flow, use case, user story map

DataWhat information exists and how does it relate?Entity model, data dictionary, state diagram

A useful way to read this topic is that Requirements Walkthrough protects requirement quality from two angles: business correctness and technical usability. Stakeholders need to know that the requirement reflects a real need, while the delivery team needs wording that can be designed, built, and tested.

Use it when

Use this topic before approval, before sprint commitment, during walkthroughs, and whenever conflicting interpretations appear.

How to apply

Ask stakeholders to confirm intent, then review requirement wording against clear, complete, consistent, feasible, necessary, and testable criteria.

Practical example

A requirement walkthrough reveals that two departments use the phrase "active provider" differently. The BA updates terminology, examples, and acceptance criteria before approval.

Common mistakes / anti-patterns

  • Confusing stakeholder agreement with requirement quality
  • Approving requirements that are not testable
  • Waiting until the final session to discover misunderstandings

Checklist

  • Confirm business need fit
  • Check clarity and consistency
  • Confirm measurability and testability
  • Resolve conflicts before approval

Key takeaway: Use Requirements Walkthrough when it helps the team make a clearer decision, reduce uncertainty, or protect business value.

# 4.14 Approval Sessions (Phiên phê duyệt)

Source page anchor: p. 133 · direct anchor

This idea matters most when Approval Sessions addresses gaining formal agreement to move forward with the requirement set. In a project setting, this topic helps the analyst convert uncertain information into a form that supports decisions, collaboration, and delivery.

Use it when

Use this topic when the team needs clearer understanding, stronger stakeholder alignment, or a more reliable basis for future work.

How to apply

Clarify the business reason, select the right participants, produce an artifact that answers a real question, and confirm the result with stakeholders.

Practical example

A BA working with product, engineering, QA, and operations records the business reason, stakeholder input, open assumptions, and next decision so the team can move forward with a shared understanding.

Common mistakes / anti-patterns

  • Documenting output without a decision purpose
  • Assuming stakeholder statements are complete
  • Skipping review with affected parties

Checklist

  • State the business reason
  • Name the affected stakeholders
  • Capture assumptions and exceptions
  • Confirm the output with the right audience

Key takeaway: Use Approval Sessions when it helps the team make a clearer decision, reduce uncertainty, or protect business value.

# 4.15 Resolve Requirements-Related Conflicts (Giải quyết xung đột liên quan đến yêu cầu)

Source page anchor: p. 134 · direct anchor

A mature team uses this topic to Resolve Requirements-Related Conflicts gives the team a structured way to move from disagreement to decision. Requirements conflict is normal when stakeholders optimize for different goals, policies, departments, budgets, or customer segments.

Use it when

Use conflict techniques when discussion repeats without progress, when authority alone would hide trade-offs, or when the decision needs transparent rationale.

How to apply

Separate facts from preferences, define decision criteria, compare options, document rationale, and identify the decision owner when consensus is not possible.

Practical example

Operations wants fewer approval steps; compliance wants stronger control. Weighted ranking compares options against risk, cycle time, audit need, and implementation effort.

Common mistakes / anti-patterns

  • Letting hierarchy replace analysis
  • Forcing consensus when escalation is needed
  • Ignoring decision criteria

Checklist

  • Separate positions from needs
  • Define criteria and constraints
  • Compare options transparently
  • Record rationale and ownership

Key takeaway: Use Resolve Requirements-Related Conflicts when it helps the team make a clearer decision, reduce uncertainty, or protect business value.
