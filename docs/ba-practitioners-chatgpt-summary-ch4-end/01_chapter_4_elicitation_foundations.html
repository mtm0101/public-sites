# Chapter 4A - Requirements Elicitation Foundations and Techniques (Chương 4A - Nền tảng và kỹ thuật khai thác yêu cầu)

Scope note: This page covers the first half of Chapter 4: how to plan, prepare, run, close, and troubleshoot elicitation. It treats elicitation as active discovery rather than passive collection, with emphasis on stakeholder readiness, questioning, listening, follow-up, and choosing techniques that fit the context.

Learning map (Bản đồ học tập)

  • Treat elicitation as discovery. A stakeholder statement is input, not automatically a requirement.
  • Choose techniques by purpose: depth, breadth, validation, observation, consensus, or creative generation.
  • Prepare objectives and questions, then adapt when the conversation reveals new information.
  • Close sessions with decisions, open issues, owners, and confirmation steps.
  • Document outputs quickly while memory and context are still fresh.

# 4.5.5 Elicitation Techniques (Các kỹ thuật khai thác)

Source page anchor: p. 77 · direct anchor

For a project team, Elicitation Techniques is a way to select or use elicitation methods so the team can uncover matching elicitation methods to information needs and stakeholder conditions. The technique is valuable when the project has uncertainty, multiple stakeholder perspectives, or practical work that is not fully captured in existing documents.

Use it when

Use this topic when the team needs a deliberate discovery method rather than an informal conversation. It is especially helpful when one source of information is incomplete or biased.

How to apply

Clarify the objective, choose participants, prepare prompts or artifacts, run the activity, record decisions and assumptions, and confirm the results with stakeholders.

Practical example

Before a workshop, the BA reviews current procedures, interviews two power users, and brings a draft process flow. The workshop then focuses on correcting assumptions and deciding exceptions instead of starting from a blank page.

Common mistakes / anti-patterns

  • Using one elicitation method for every situation
  • Inviting only loud or available stakeholders
  • Skipping confirmation after the session

Checklist

  • Clarify objective before the activity
  • Select participants intentionally
  • Prepare materials and questions
  • Confirm notes, decisions, and follow-up actions
TechniqueBest fitWatch out for
BrainstormingGenerating many possible needs, risks, causes, or solutionsJudging ideas too early
Document analysisLearning the current state before using stakeholder timeAssuming documents reflect real practice
WorkshopsBuilding shared agreement and resolving cross-functional issuesLetting authority dominate evidence
ObservationFinding tacit work, shortcuts, and exceptionsChanging behavior because people know they are watched
PrototypingClarifying uncertain user interactions or report expectationsStakeholders confusing prototype with final design
SurveysReaching many respondents with consistent questionsUsing survey results without follow-up context
